## 1. 로드밸런싱 개요

### 1-1. 로드밸런싱이란?

로드 밸런서(LB)를 이용하여 하나의 서비스 요청을 여러 서버로 분산해, **성능과 가용성**을 높이는 트래픽 제어 기술

- 사용자는 한 주소로 접속하지만, 내부적으로는 여러 서버가 나눠 처리
- 트래픽의 입구에서 병목을 줄이고 장애를 격리
- 트래픽 분산뿐만 아니라 보안 역할도 수행

### 1-2. 로드밸런싱이 필요한 이유

- 트래픽은 예측보다 **급증**이 더 위험
- 단일 서버는 병목(지연 증가) → 장애 전파(서비스 다운)로 이어짐
- 서버가 죽어도 사용자 경험은 유지되도록 함 (Health Check 기반 우회)

### 1-3. 로드밸런서의 종류

| 종류 | 계층 | 특징 |
|------|------|------|
| **ALB** (Application) | L7 | HTTP 요청 기반 분산, 비정상 트래픽 필터링 가능 |
| **NLB** (Network) | L4 | TCP/UDP 단위 분산, 빠르고 저렴 |
| **GLB** (Gateway) | L3/L4 | 보안 장비로 트래픽을 투명하게 전달, 보안 검사 목적 |

## 2. Nginx 기반 로드밸런싱

### 2-1. 단일 서버 vs 3대 분산

| 항목 | 단일 서버 | 3대 분산 (Scale-out) |
|------|----------|---------------------|
| RPS | ~600 | 서버당 ~260 (총 ~780) |
| CPU 사용률 | 40~65% | 15~35% |

→ 서버를 3대로 늘리고 로드밸런서를 추가한 결과, **총 처리량 약 30% 증가**, 개별 서버 부하 **절반 이하로 감소**

### 2-2. 알고리즘별 비교 (과부하 서버 1대 + 기본 서버 2대 시나리오)

K6 부하 테스트 — VU 1,000명 (30초 증가 / 2분 유지 / 10초 정리)

| 알고리즘 | 동작 방식 | 장점 | 단점 |
|----------|----------|------|------|
| **Round Robin** | 순서대로 기계적 분배 | 설정 간단, 동일 스펙 환경에서 효율적 | 서버 성능 차이 시 과부하 발생 가능 |
| **Least Connections** | 현재 연결 수가 가장 적은 서버로 전달 | 실시간 상태 반영, 이질적 환경에 강함 | 연결 수 계산 오버헤드 |
| **Weighted Round Robin** | 가중치 비율에 따라 차등 분배 | 스펙 차이가 명확할 때 유용 | 가중치 예측 실패 시 비효율 |

**실험 결과 요약:**
- **Round Robin** — 과부하 서버(🟡)도 동일 요청을 받아 CPU 40~57%까지 상승, 서버 다운 위험
- **Least Connections** — 과부하 서버는 ~120 RPS, 정상 서버는 ~400 RPS로 자동 조절, CPU 모두 20~30%대 안정
- **Weighted RR (3:2:1)** — 의도대로 분배되나, 가중치 예측이 틀리면 오히려 비효율 발생

## 3. AWS 기반 로드밸런싱

### 3-1. 분산 효과 검증

| 구성 | 서버당 트래픽 |
|------|-------------|
| 서버 1대 | ~120,000 bytes/s |
| 서버 4대 (ALB) | ~32,000~33,000 bytes/s |

→ 서버 수만 늘려도 서버당 부하는 **선형적으로 감소**

### 3-2. 헬스체크 기반 장애 격리

- 서버 1대 중지 시 ALB가 **unhealthy 상태로 판단 → 즉시 제외 → 나머지 서버로 트래픽 자동 재분산**
- 서버 재시작 시 다시 healthy로 복귀하여 트래픽 재배분

### 3-3. 로드밸런싱 유무에 따른 장애 조치 비교

부하 테스트: VU 200명 (30초 증가 / 2분 유지 / 30초 정리)

| 구분 | 실패율 |
|------|--------|
| LB 미적용 | **7.49%** |
| LB 적용 (ALB) | **4.52%** |

→ 장애 서버를 감지하고 트래픽을 끊어내며, 살아있는 서버로 요청을 전환하여 **서비스 무중단 유지**

---

## 4. 종합 정리 및 고찰

**Scale-out의 효과** — 단일 서버 CPU 40~65% → 3대 분산 시 15~35%로 병목 완화

**알고리즘 선택** — '정답 알고리즘'은 없고, 서버 스펙·트래픽 패턴에 따라 적절히 선택해야 함

**AWS 고찰** — 분산은 "성능", 헬스체크는 "가용성"을 만들며, 멀티 AZ 구성으로 장애 범위 확장 대응 가능

> *"로드밸런싱은 서버를 늘리는 기술이 아니라, 트래픽을 올바르게 분산하고 비정상 타깃을 자동 제외해 성능과 가용성을 동시에 확보하는 운영 기술이다."*
